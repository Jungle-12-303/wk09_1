## 슬라이드 1: "Interrupt는 대체 뭔데?"

### 슬라이드 내용

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│          Interrupt는 대체 뭔데?                            │
│                                                          │
│     — 스레드를 공부하는데, 인터럽트가 왜 나와?              │
│                                                          │
│                                                          │
│   main 스레드가 코드를 실행하고 있다.                       │
│   그런데 갑자기 timer_interrupt라는 함수가 끼어든다.         │
│                                                          │
│   "CPU가 하나인데, 이게 어떻게 가능하지?"                    │
│                                                          │
│   혹시 타이머 전용 스레드가 따로 있는 건가?                  │
│   Pintos가 내부적으로 뭔가 몰래 돌리고 있나?                │
│                                                          │
│                                                          │
│   → 아닙니다.                                             │
│     소프트웨어가 아니라 하드웨어가 CPU를 강제로 멈춥니다.     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 발표 스크립트

> 저는 Pintos 코드를 처음 읽을 때 가장 혼란스러웠던 게 이거였습니다.
>
> "스레드를 공부하는 건데, timer_interrupt라는 함수가 계속 나온다.
> 그런데 main 스레드가 뭔가 실행하고 있는 도중에 이게 끼어든다.
> CPU가 하나인데, 이게 어떻게 가능한 거지?"
>
> 처음에는 Pintos가 내부적으로 타이머 전용 스레드를 따로 돌리나 싶었습니다.
> 하지만 아니었습니다.
>
> 정답은 — 소프트웨어가 아니라 하드웨어가 끼어드는 겁니다.
> CPU 옆에 8254라는 타이머 칩이 있고, 이 칩이 일정 주기마다
> CPU에 전기 신호를 보냅니다. CPU는 지금 뭘 하고 있든
> 그 신호를 받으면 강제로 멈추고 미리 등록된 핸들러로 점프합니다.
>
> 이걸 이해하고 나니까 Pintos 전체가 보이기 시작했습니다.
> 다음 장에서 왜 이런 설계가 필요한지 보겠습니다.

## 슬라이드 2: "왜 하드웨어 타이머가 필요할까?"

### 슬라이드 내용

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│       왜 하드웨어 타이머가 필요할까?                        │
│                                                          │
│                                                          │
│   이런 코드가 실행된다고 생각해보자:                         │
│                                                          │
│   ┌─────────────────────────────────┐                    │
│   │  void stuck_thread (void) {     │                    │
│   │      while (1) {                │                    │
│   │          do_something();        │                    │
│   │      }                          │                    │
│   │  }                              │                    │
│   └─────────────────────────────────┘                    │
│                                                          │
│                                                          │
│   ┌───────────────────────────────────────────────────┐  │
│   │                                                   │  │
│   │   CPU:  명령어 실행 ──▶ 다음 명령어 ──▶ ...       │  │
│   │             ▲                     │               │  │
│   │             └─────────────────────┘               │  │
│   │         CPU는 이게 무한루프인지 모른다              │  │
│   │                                                   │  │
│   └───────────────────────────────────────────────────┘  │
│                                                          │
│   • CPU는 그냥 다음 명령어를 실행할 뿐이다                  │
│   • 무한루프인지, 정상인지 판단할 능력이 없다                │
│   • 잠든 스레드는 스스로 깨어날 수 없다                     │
│                                                          │
│           ┌──────────┐    IRQ0     ┌──────┐             │
│           │ 8254 PIT │ ─────────▶  │ CPU  │             │
│           │ (초시계)  │   100Hz    │      │             │
│           └──────────┘            └──────┘             │
│                                                          │
│   → 외부 하드웨어가 주기적으로 CPU를 "깨워야" 한다          │
│   → Pintos에서는 이것이 timer_interrupt()                  │
│                                                          │
│   ※ QEMU가 이 하드웨어를 소프트웨어로 시뮬레이션             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 발표 스크립트

> CPU 입장에서 생각해봅시다.
>
> 어떤 스레드가 이런 코드를 실행합니다. while(1) 무한루프입니다.
> CPU는 그냥 주어진 명령어를 순서대로 실행할 뿐입니다.
> 지금 실행 중인 코드가 무한루프인지, 정상적인 건지 CPU 스스로는 알 수 없습니다.
>
> 잠들어 있는 스레드도 마찬가지입니다. 스스로 깨어날 방법이 없습니다.
> 누군가가 주기적으로 "야, 지금 깨어날 시간이야" 하고 끼어들어야 합니다.
>
> 그래서 CPU 옆에 하드웨어 초시계를 달아놓은 겁니다.
> 8254 PIT라는 타이머 칩이 초당 100번 IRQ0 신호를 CPU에 보내고,
> CPU는 이 신호를 받으면 하던 일을 멈추고 커널의 timer_interrupt 핸들러로 점프합니다.
>
> 그러면 이 timer_interrupt 안에서 구체적으로 뭘 하길래
> 스레드가 전환되고, 잠든 스레드가 깨어나는 걸까요?
> 다음 장에서 전체 흐름을 보겠습니다.

## 슬라이드 3: "CPU 입장에서의 인터럽트 흐름"

### 슬라이드 내용

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   CPU 입장에서의 인터럽트 흐름                                            │
│                                                                         │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                    ★ 인터럽트 사이클 ★                           │   │
│   │                                                                 │   │
│   │   RUNNING 스레드가 코드를 실행하고 있다                           │   │
│   │       │                                                         │   │
│   │       │  ◀── 8254 PIT: IRQ0 (매 tick, 100Hz)                    │   │
│   │       ▼                                                         │   │
│   │   intr_entry (어셈블리)                                          │   │
│   │       │  레지스터 15개 + 세그먼트 저장                             │   │
│   │       ▼                                                         │   │
│   │   intr_handler()                                                │   │
│   │       │                                                         │   │
│   │       ▼                                                         │   │
│   │   timer_interrupt()                                              │   │
│   │       ├── ticks++                                               │   │
│   │       ├── thread_tick()                                          │   │
│   │       │     └── 4틱마다 → intr_yield_on_return() (플래그 설정)    │   │
│   │       │                                                         │   │
│   │       └── thread_awake(ticks)                                    │   │
│   │             └── sleep_list 순회                                   │   │
│   │                   wakeup_tick ≤ ticks인 스레드 → thread_unblock() │   │
│   │                                                                 │   │
│   │   intr_handler 복귀:                                              │   │
│   │       ├── pic_end_of_interrupt()  ← 먼저 EOI 전송                │   │
│   │       └── yield_on_return 플래그 확인                              │   │
│   │             └── true → thread_yield() → schedule()               │   │
│   │                                                                 │   │
│   │   iretq (어셈블리) → 레지스터 복원 → 원래 코드로 복귀               │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │              ★ 스레드 상태와 대기 큐 ★                            │   │
│   │                                                                 │   │
│   │                thread_create()                                   │   │
│   │                     │                                           │   │
│   │                     ▼                                           │   │
│   │  ┌─────────┐    ┌─────────┐  schedule()  ┌──────────┐          │   │
│   │  │ BLOCKED │───▶│  READY  │────────────▶│ RUNNING  │          │   │
│   │  └─────────┘    └─────────┘             └──────────┘          │   │
│   │       ▲              ▲                       │                 │   │
│   │       │              └──── thread_yield() ◀──┘                 │   │
│   │       │                                      │                 │   │
│   │       └────── thread_block() ◀───────────────┘                 │   │
│   │                                                                 │   │
│   │                                                                 │   │
│   │  BLOCKED 상태의 스레드들이 대기하는 곳:                           │   │
│   │                                                                 │   │
│   │  ┌─────────────┐  ┌──────────────┐  ┌────────────────────┐     │   │
│   │  │ sleep_list  │  │ sema.waiters │  │ cond.waiters       │     │   │
│   │  │             │  │              │  │  └─ sema.waiters   │     │   │
│   │  │ wakeup_tick │  │ lock_acquire │  │                    │     │   │
│   │  │ 기준 대기   │  │ sema_down    │  │ cond_wait          │     │   │
│   │  └──────┬──────┘  └──────┬───────┘  └────────┬───────────┘     │   │
│   │         │                │                   │                 │   │
│   │         │ thread_awake() │ sema_up()         │ cond_signal()   │   │
│   │         │ (매 tick)      │                   │                 │   │
│   │         ▼                ▼                   ▼                 │   │
│   │      ┌──────────────────────────────────────────┐              │   │
│   │      │              ready_list                   │              │   │
│   │      │         (우선순위 내림차순 정렬)             │              │   │
│   │      └─────────────────────┬────────────────────┘              │   │
│   │                            │ schedule()                        │   │
│   │                            ▼                                   │   │
│   │                     ┌──────────┐                               │   │
│   │                     │ RUNNING  │                               │   │
│   │                     └──────────┘                               │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 발표 스크립트

> 이 슬라이드는 두 부분으로 나뉩니다.
>
> 위쪽은 인터럽트 사이클입니다.
> RUNNING 스레드가 코드를 실행하다가 8254 PIT가 IRQ0을 보내면,
> CPU는 intr_entry 어셈블리로 점프합니다. 여기서 레지스터 15개와
> 세그먼트 레지스터를 스택에 저장합니다.
>
> 그 다음 intr_handler가 timer_interrupt를 호출합니다.
> timer_interrupt 안에서 ticks를 증가시키고, thread_tick()을 호출합니다.
> thread_tick은 4틱이 지났으면 intr_yield_on_return 플래그를 세팅합니다.
> 왜 바로 yield를 안 하냐면 — 인터럽트 핸들러 안에서는 EOI를 아직 안 보냈기 때문입니다.
>
> 그 다음 thread_awake()가 sleep_list를 순회하면서
> 깨어날 시간이 된 스레드들을 unblock합니다.
>
> 인터럽트 처리가 끝나면 pic_end_of_interrupt로 EOI를 보내고,
> yield 플래그가 켜져있으면 그제서야 thread_yield를 호출합니다.
>
> 아래쪽은 스레드 상태와 대기 큐입니다.
> BLOCKED 스레드들은 sleep_list, 세마포어의 waiters, 컨디션의 waiters에 나뉘어 대기합니다.
> 각각 thread_awake, sema_up, cond_signal에 의해 ready_list로 올라오고,
> schedule()이 호출되면 가장 우선순위 높은 스레드가 RUNNING이 됩니다.


### 발표 스크립트

> 앞 슬라이드의 인터럽트 사이클이 실제로 일어나는지 GDB로 증명해봤습니다.
>
> priority-donate-one 테스트에서 main 스레드가 line 35의 msg()를
> 실행하는 순간에 breakpoint를 잡았습니다.
>
> 장면 1: RIP가 0x8004218e5b입니다. test 코드 영역이죠.
> 이건 슬라이드 3에서 "RUNNING 스레드가 코드를 실행" 하는 지점입니다.
>
> 장면 2: 타이머 인터럽트에서 잡았더니 RIP가 0x800420dd75로 바뀌어 있습니다.
> 완전히 다른 주소입니다. timer_interrupt 함수 안이죠.
> 그런데 thread_current()->name을 찍으면 여전히 "main"입니다.
> main 스레드 컨텍스트인데 CPU는 인터럽트 핸들러를 실행하고 있는 겁니다.
> 슬라이드 3에서 "timer_interrupt()" 지점에 해당합니다.
>
> 장면 3: 인터럽트가 끝나고 line 37에서 잡았더니
> RIP가 0x8004218e8c로 돌아왔습니다.
> 슬라이드 3의 "iretq → 원래 코드로 복귀" 지점입니다.
>
> 0x800421 대역에서 0x800420 대역으로 갔다가 다시 0x800421로 —
> CPU가 실제로 점프했다는 걸 주소로 볼 수 있습니다.

## 슬라이드 5: "Page Fault — 부팅 순서를 몰라서 생긴 버그"

### 슬라이드 내용

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   Page Fault — 부팅 순서를 몰라서 생긴 버그                               │
│                                                                         │
│                                                                         │
│   ── 버그 1: check_preemption에서 Page Fault ──                          │
│                                                                         │
│   thread_create 끝에 check_preemption()을 넣었더니                        │
│   부팅할 때 Page Fault로 터졌다.                                          │
│                                                                         │
│   원인: 부팅 초기에 idle 스레드를 만들 때                                   │
│         ready_list가 비어있는 상태에서 list_front() 호출                    │
│                                                                         │
│   ┌─────────────────────────────────────────────┐                       │
│   │  void                                       │                       │
│   │  check_preemption (void) {                  │                       │
│   │      if (list_empty (&ready_list))   ← 가드 │                       │
│   │          return;                            │                       │
│   │      struct thread *front =                 │                       │
│   │          list_entry (list_front (&ready_list),                      │
│   │                      struct thread, elem);  │                       │
│   │      if (front->priority                    │                       │
│   │              > thread_current ()->priority) {│                       │
│   │          if (intr_context ())               │                       │
│   │              intr_yield_on_return ();        │                       │
│   │          else                               │                       │
│   │              thread_yield ();               │                       │
│   │      }                                      │                       │
│   │  }                                          │                       │
│   └─────────────────────────────────────────────┘                       │
│                                                                         │
│   list_empty 가드 한 줄로 해결.                                           │
│   또한 intr_context() 분기도 중요하다:                                    │
│   인터럽트 안이면 직접 yield 대신 플래그만 세팅한다.                        │
│                                                                         │
│                                                                         │
│   ── 버그 2: priority가 안 내려가는 문제 ──                                │
│                                                                         │
│   thread_set_priority에서 현재보다 낮은 우선순위로                         │
│   변경하는 것이 무시되는 버그.                                             │
│                                                                         │
│   ┌─────────────────────────────────────────────┐                       │
│   │  /* 수정 전 (버그) */                         │                       │
│   │  void                                       │                       │
│   │  thread_set_priority (int new_priority) {   │                       │
│   │      if (new_priority > curr->priority)     │                       │
│   │          curr->priority = new_priority;     │                       │
│   │  }                                          │                       │
│   │                                             │                       │
│   │  /* 수정 후 */                                │                       │
│   │  void                                       │                       │
│   │  thread_set_priority (int new_priority) {   │                       │
│   │      curr->origin_priority = new_priority;  │                       │
│   │      refresh_priority (curr);               │                       │
│   │      check_preemption ();                   │                       │
│   │  }                                          │                       │
│   └─────────────────────────────────────────────┘                       │
│                                                                         │
│   origin_priority에 저장 후 refresh_priority로 재계산.                     │
│   donation이 있으면 기부 중 최댓값과 비교하여 결정한다.                     │
│                                                                         │
│   ┌─────────────────────────────────────────────┐                       │
│   │  void                                       │                       │
│   │  refresh_priority (struct thread *t) {      │                       │
│   │      t->priority = t->origin_priority;      │                       │
│   │      for (e = list_begin (&t->donations);   │                       │
│   │           e != list_end (&t->donations);    │                       │
│   │           e = list_next (e)) {              │                       │
│   │          struct thread *donor =             │                       │
│   │              list_entry (e, struct thread,   │                       │
│   │                          donation_elem);    │                       │
│   │          if (donor->priority > t->priority) │                       │
│   │              t->priority = donor->priority; │                       │
│   │      }                                      │                       │
│   │  }                                          │                       │
│   └─────────────────────────────────────────────┘                       │
│                                                                         │
│   → 인터럽트 타이밍, 부팅 순서, 기부 구조를 모르면                         │
│     생각지 못한 곳에서 터진다.                                             │
│   → 코드를 직접 읽는 과정은 생략할 수 없다.                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 발표 스크립트

> 마지막으로, 인터럽트 구조를 몰라서 겪었던 버그 두 가지를 소개합니다.
>
> 첫 번째는 Page Fault입니다.
> thread_create 마지막에 "새 스레드가 나보다 높으면 양보해라"는
> check_preemption을 추가했는데, 부팅할 때 터졌습니다.
> 부팅 초기에 idle 스레드를 만들 때 ready_list가 비어있어서
> list_front()가 빈 리스트에 접근한 겁니다.
> list_empty 가드 한 줄로 해결했습니다.
>
> 이 함수에서 또 중요한 건 intr_context 분기입니다.
> 인터럽트 핸들러 안에서 호출되면 직접 yield 대신
> 플래그만 세팅합니다. 앞에서 본 EOI 문제 때문이죠.
>
> 두 번째는 thread_set_priority에서 우선순위가 안 내려가는 버그입니다.
> 처음에는 new_priority가 현재보다 높을 때만 바꿨습니다.
> 이러면 높아진 건 반영되는데 낮아진 건 무시됩니다.
>
> 수정 후에는 origin_priority에 저장하고 refresh_priority로 재계산합니다.
> refresh_priority는 원래 우선순위에서 시작해서
> donations 리스트의 기부자 중 최댓값과 비교합니다.
> 기부가 있으면 높은 쪽을, 없으면 origin_priority를 씁니다.
>
> 인터럽트 타이밍, 부팅 순서, 기부 구조 — 이런 것들을 모르면
> 생각지 못한 곳에서 터집니다.
> 결국 코드를 직접 읽는 과정은 생략할 수 없다는 게
> 이번 주의 가장 큰 깨달음이었습니다.
