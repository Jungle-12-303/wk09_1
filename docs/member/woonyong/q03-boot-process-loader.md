# Q03. 부팅 전 과정과 loader.S의 역할 -- 부트 섹터, A20, GDT, IDE PIO, 커널 로드

> Pintos threads/loader.S, threads/start.S | 중급

## 질문

1. 컴퓨터 전원이 켜지면 커널 코드가 실행되기까지 어떤 과정을 거치는가
2. BIOS가 디스크 첫 섹터를 0x7C00에 올리는데, 왜 하필 0x7C00인가
3. loader.S가 512바이트인 이유는 무엇이고, 이 512바이트에는 무엇이 들어 있는가
4. cli(인터럽트 비활성화)는 CPU 연산을 막는 것인가
5. A20 게이트, E820 메모리 맵, 보호 모드 전환은 각각 왜 필요한가
6. loader.S가 커널을 "올린다"는 것은 구체적으로 무엇을 어디에 복사하는 것인가

## 답변

### 최우녕

> 컴퓨터 전원이 켜지면 커널 코드가 실행되기까지 어떤 과정을 거치는가

CPU는 전원이 켜지면 아무것도 모르는 상태다. OS도 없고 RAM에 뭐가 있는지도 모른다.
유일하게 아는 건 BIOS 펌웨어의 코드를 실행하라는 것뿐이다.

커널은 디스크에 있고, CPU는 디스크에서 직접 코드를 실행할 수 없다. 반드시 RAM에 올려야 한다.
문제는 BIOS가 딱 512바이트만 읽을 줄 안다는 것이다. 그래서 2단계로 나뉜다:

1. BIOS가 512바이트짜리 작은 프로그램(loader.S)을 RAM에 올리고 실행
2. 그 작은 프로그램이 나머지 커널 전체를 RAM에 올리고 실행

전체 흐름:

```
[전원 ON]
    │
    ▼
[BIOS] "디스크 섹터 0 → RAM 0x7C00 복사, 거기로 점프"
    │
    ▼
[loader.S] (512바이트, 16비트)
    ├─ A20 열기 (1MB 이상 접근 가능하게)
    ├─ 메모리 지도 받기 (E820)
    ├─ 32비트 모드로 전환 (GDT + CR0)
    ├─ 디스크 섹터 1~N → RAM 1MB에 복사 (커널 전체)
    └─ RAM 1MB로 점프
          │
          ▼
[start.S] (32비트 → 64비트 전환)
    └─ main() 호출
          │
          ▼
[main()] (C 코드)
    └─ thread_init() → 스레드 시스템 시작
```

각 단계가 없으면 다음 단계가 불가능하다.

> BIOS가 디스크 첫 섹터를 0x7C00에 올리는데, 왜 하필 0x7C00인가

IBM이 1981년에 정한 약속이다. 기술적으로 깊은 이유가 있는 건 아니다.

당시 IBM PC의 최소 RAM이 32KB(= 0x8000)였다.
부트 섹터(512바이트)를 올리고 아래에 스택 공간도 확보하려면 적당한 위치가 필요했다.
0x7C00(= 31KB 지점)에 올리면 아래쪽에 스택용 약 31KB, 위쪽에 약 1KB 여유가 생겨서 합리적이었다.
그 뒤로 모든 PC BIOS가 이 주소를 쓰게 되었고, 지금도 그대로다.

> loader.S가 512바이트인 이유는 무엇이고, 이 512바이트에는 무엇이 들어 있는가

BIOS는 디스크의 첫 번째 섹터(= 512바이트)만 읽도록 설계되어 있다.
loader.S 전체가 컴파일되면 정확히 512바이트가 된다.

```
loader.S →(어셈블)→ loader.bin (정확히 512바이트)
이 loader.bin이 디스크 이미지(os.dsk)의 섹터 0에 들어감
```

512바이트를 강제하는 코드:

```asm
.org LOADER_SIG - LOADER_BASE   ← 510번째 바이트 위치로 이동
.word 0xaa55                     ← BIOS 시그니처 (이게 없으면 부팅 거부)
```

이 512바이트 안에 들어 있는 것:

| 내용 | 역할 |
|------|------|
| A20 게이트 활성화 코드 | 1MB 이상 메모리 접근 허용 |
| E820 메모리 맵 조회 코드 | RAM 사용 가능 영역 파악 |
| GDT + 보호 모드 전환 코드 | 32비트 모드로 업그레이드 |
| IDE PIO 디스크 읽기 코드 | 커널을 RAM에 복사 |
| 커맨드라인 인자 영역 | pintos -- -q run alarm-multiple 등의 인자 저장 |
| 0xAA55 시그니처 | BIOS가 부트 섹터로 인식하기 위한 매직 넘버 |

> cli(인터럽트 비활성화)는 CPU 연산을 막는 것인가

아니다. CPU는 계속 코드를 실행한다.

인터럽트란 "지금 하던 일을 잠깐 멈추고 이 급한 일을 처리해"라는 신호다.
예를 들어 키보드를 누르면 키보드가 CPU에게 인터럽트를 보내고, CPU는 하던 일을 멈추고 키보드 처리 코드(핸들러)로 점프한다.

부팅 직후에는 핸들러가 아직 없다.
이 상태에서 인터럽트가 오면 CPU가 "핸들러 찾아야 하는데… 없네?" 하고 크래시한다.
그래서 cli로 "인터럽트 신호가 와도 무시해"라고 설정한다.
나중에 커널이 핸들러를 다 등록한 뒤에 sti로 다시 켠다.

```
cli = Clear Interrupt flag → 인터럽트 무시 (CPU는 계속 실행)
sti = Set Interrupt flag   → 인터럽트 다시 수신
```

> A20 게이트, E820 메모리 맵, 보호 모드 전환은 각각 왜 필요한가

**A20 게이트 — "잠긴 문 열기"**

1981년 IBM PC 호환 문제로, 부팅 시 CPU의 20번째 주소선이 꺼져 있다.
A20이 꺼지면 주소 비트 20이 항상 0으로 강제되어 1MB 이상 메모리에 접근이 불가능하다.
커널을 1MB(0x100000) 위치에 올려야 하니까, 이 차단을 해제해야 한다.

키보드 컨트롤러(포트 0x64, 0x60)를 통해 해제한다:

```asm
outb $0xd1, $0x64   // "출력 포트 쓰기" 명령
outb $0xdf, $0x60   // 0xDF = A20 활성화 비트 포함
```

**E820 메모리 맵 — "건물 도면 받기"**

RAM의 모든 영역이 다 쓸 수 있는 건 아니다.
비디오 메모리, BIOS 자체가 차지하는 영역 등이 있다.
BIOS 인터럽트(INT 15h / E820h)로 "여기는 써도 됨 / 여기는 예약됨"이라는 지도를 받는다.
이 지도를 나중에 palloc_init()이 읽어서 페이지 할당기를 세팅한다.

각 엔트리는 24바이트:

| 오프셋 | 크기 | 내용 |
|--------|------|------|
| +0 | 8B | 영역 시작 주소 |
| +8 | 8B | 영역 크기 |
| +16 | 4B | 타입 (1=사용 가능, 2=예약) |
| +20 | 4B | ACPI 확장 속성 |

**보호 모드 전환 — "도구 업그레이드"**

16비트 리얼 모드에서는 주소를 최대 1MB까지밖에 표현 못 한다.
커널을 1MB 위치에 올리고 접근하려면 32비트 모드가 필요하다.

```asm
lgdt gdtdesc        // GDT를 CPU에 등록
orl $CR0_PE, %eax   // CR0.PE = 1 → 보호 모드 활성화
ljmp $SEL_KCSEG, $protcseg  // far jump로 32비트 코드 세그먼트 진입
```

> loader.S가 커널을 "올린다"는 것은 구체적으로 무엇을 어디에 복사하는 것인가

디스크(os.dsk)의 섹터 1~N에 있는 kernel.bin을 RAM 1MB(0x100000) 위치에 복사하는 것이다.
kernel.bin에는 start.S, thread.c, init.c, synch.c 등 모든 커널 코드가 컴파일·링크되어 들어 있다.

```
디스크(os.dsk)                         RAM
┌──────────────┐                ┌──────────────────┐
│ 섹터 0       │ (loader.S)     │ 0x00000 IVT/BIOS │
│ 섹터 1       │──────┐        │ 0x07C00 loader.S  │ ← BIOS가 올림
│ 섹터 2       │      │        │         ...       │
│ ...          │      │        │                   │
│ 섹터 N       │      └──────→ │ 0x100000 커널     │ ← loader.S가 올림
│              │               │ (start.S+thread.c │
│              │               │  +init.c+synch.c  │
│              │               │  +... 전부)       │
└──────────────┘               └──────────────────┘
```

OS가 없으니까 파일 시스템도 없고 디스크 드라이버도 없다.
IDE 컨트롤러 하드웨어 포트(0x1F0~0x1F7)에 직접 명령을 보내서 512바이트씩 RAM으로 복사한다.

```asm
movl $1, %ebx              // 시작 섹터 = 1 (섹터 0은 부트 섹터)
movl $LOADER_PHYS_BASE, %edi  // 목적지 = 0x100000 (1MB)

// 반복:
//   0x1F7 폴링 → BSY 해제 대기
//   0x1F2에 1 → "1섹터 읽겠다"
//   0x1F3~0x1F6에 LBA 주소 설정
//   0x1F7에 0x20 → "읽기 명령"
//   DRQ 대기 → 데이터 준비되면
//   rep insw → 256워드(=512바이트) 읽어서 EDI(RAM)에 복사
//   총 KERNEL_LOAD_PAGES × 8 섹터 반복
```

로드가 끝나면 1MB 주소로 점프하여 start.S가 실행된다.

```asm
movl $LOADER_PHYS_BASE, %eax
call *%eax   // → start.S의 bootstrap → 64비트 전환 → main()
```

### 수치 검증

부팅 시 메모리 레이아웃을 주소 단위로 확인한다:

| 주소 범위 | 크기 | 내용 |
|-----------|------|------|
| 0x00000 ~ 0x003FF | 1KB | IVT (리얼 모드 인터럽트 벡터) |
| 0x00400 ~ 0x004FF | 256B | BDA (BIOS Data Area) |
| 0x07C00 ~ 0x07DFF | 512B | 부트 섹터 (loader.S) |
| 0x07E00 ~ 0x0FFFF | ~33KB | 부트 스택 등 자유 영역 |
| 0x10000 ~ | 가변 | E820 메모리 맵 데이터 |
| 0x100000 (1MB) ~ | 가변 | 커널 (loader.S가 올림) |

0x7C00 = 31,744 (약 31KB)
0x100000 = 1,048,576 (정확히 1MB)

1섹터 = 512바이트, 1페이지 = 4KB = 8섹터
KERNEL_LOAD_PAGES 페이지 × 8섹터 = 디스크에서 읽는 총 섹터 수

0xAA55 시그니처 위치: 부트 섹터의 바이트 510~511 (0-based)
512 - 2 = 510번째 바이트부터 2바이트
