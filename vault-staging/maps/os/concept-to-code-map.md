# 개념에서 실제 코드까지 가는 OS 지도

## 이 지도의 목적

이 문서는 OS 학습의 0레벨 지도다.

목표는 초보 개발자가 작은 질문 하나에서 시작해 다음 순서로 자연스럽게 내려가게 만드는 것이다.

- 추상 개념
  - 왜 필요한가
  - 어떤 문제를 해결하는가
- 실제 OS
  - Linux / Windows에서는 어떻게 동작하는가
  - 실제 OS가 PintOS보다 더 복잡한 이유는 무엇인가
- PintOS
  - 과제에서는 어떤 코드와 구조체로 축소되어 나타나는가
  - 어디를 고치고 어디를 관찰해야 하는가
- QEMU
  - PintOS가 믿는 CPU, 메모리, 인터럽트, 장치는 QEMU에서 어떻게 에뮬레이션되는가
- 숫자와 바이트
  - 주소, 레지스터, 오프셋, 페이지, tick, fd를 실제 값으로 계산한다
- 디버깅
  - GDB, 메모리 덤프, breakpoint로 눈으로 확인한다

## 큰 질문 트리

- Q0. 실제 OS, PintOS, QEMU는 각각 무엇인가?
  - 먼저 볼 문서
    - [[real-os-pintos-qemu-differences]]
  - 이해해야 할 결론
    - Linux / Windows는 실제 운영체제다.
    - PintOS는 교육용 OS다.
    - QEMU는 OS가 아니라 하드웨어를 흉내 내는 host process다.
  - 코드로 확인할 곳
    - PintOS: `pintos/userprog/syscall.c`
    - QEMU: `target/i386/gdbstub.c`, `system/physmem.c`, `hw/intc/apic.c`
- Q1. CPU 레지스터 값은 실제로 어디에 있는가?
  - 먼저 볼 문서
    - [[cpu-register-execution]]
  - 핵심 키워드
    - `RIP`
    - `RSP`
    - `RAX`
    - `RDI`
    - `RSI`
    - `RDX`
    - `CR3`
    - `RFLAGS`
  - 실제 OS 관점
    - Linux / Windows는 context switch, syscall, interrupt 때 레지스터 상태를 저장/복원한다.
  - PintOS 관점
    - `struct intr_frame`에 interrupt/syscall 진입 시점의 레지스터 snapshot이 담긴다.
  - QEMU 관점
    - guest register는 QEMU의 `CPUX86State` 같은 guest CPU state로 표현된다.
  - 숫자로 확인
    - `write(1, 0x8048123, 5)`에서 `RDI=1`, `RSI=0x8048123`, `RDX=5`
- Q2. 운영체제는 실행 중인 코드를 어떻게 멈추고 다시 CPU를 얻는가?
  - 먼저 볼 문서
    - [[interrupt-timer-qemu]]
    - [[thread-scheduler-trace]]
    - [[context-switch-trace]]
    - [[week-1-threads-map]]
    - [[pintos-intrusive-list-lab]]
  - 핵심 키워드
    - timer interrupt
    - preemption
    - `TIME_SLICE`
    - `ready_list`
    - `schedule`
    - context switch
  - 실제 OS 관점
    - Linux / Windows는 APIC, per-CPU run queue, priority, fairness, SMP, timer tick 또는 tickless 정책을 다룬다.
  - PintOS 관점
    - `timer_interrupt` -> `thread_tick` -> `intr_yield_on_return` -> `schedule`
    - 교육용으로 ready list와 단순 scheduler를 중심으로 본다.
  - QEMU 관점
    - 실제 타이머 장치가 아니라 QEMU의 가상 장치와 interrupt controller가 guest interrupt를 만든다.
  - 숫자로 확인
    - `TIME_SLICE = 4`
    - 4 ticks마다 현재 스레드가 CPU를 계속 쓸 수 있는지 검사한다.
- Q3. 유저 프로그램은 어떻게 커널 기능을 요청하는가?
  - 먼저 볼 문서
    - [[syscall-end-to-end]]
    - [[argument-passing-lab]]: exec 이후 “프로세스 시작 인자”가 스택에 쌓이는 방식
    - [[week-2-user-programs-map]]
    - [[file-descriptor-knowledge]]: fd/handle이 커널 객체를 가리키는 방식
  - 핵심 키워드
    - user mode
    - kernel mode
    - syscall ABI
    - file descriptor
    - [[user-pointer-validation-trace]]
    - `struct intr_frame`
  - 실제 OS 관점
    - Linux는 fd 중심의 syscall ABI를 제공한다.
    - Windows는 Win32 API와 handle 중심 모델을 제공한다.
  - PintOS 관점
    - `syscall_handler`가 `f->R.rax`로 syscall 번호를 보고, `f->R.rdi/rsi/rdx`로 인자를 읽는다.
  - QEMU 관점
    - QEMU는 syscall 의미를 처리하지 않는다. guest CPU transition과 instruction 실행을 재현한다.
  - 숫자로 확인
    - `write(1, 0x8048123, 5)`
      - fd: `1`
      - user buffer: `0x8048123`
      - size: `5`
- Q4. 가상 주소는 어떻게 실제 바이트가 되는가?
  - 먼저 볼 문서
    - [[address-translation-memory]]
    - [[week-3-4-virtual-memory-map]]
  - 핵심 키워드
    - VA
    - PA
    - VPN
    - offset
    - PTE
    - TLB
    - CR3
    - page fault
    - supplemental page table
  - 실제 OS 관점
    - Linux / Windows는 demand paging, COW, mmap, swap, page cache, protection을 모두 포함한다.
  - PintOS 관점
    - `page_fault()`가 `rcr2()`로 fault address를 읽고 `vm_try_handle_fault()`로 넘긴다.
    - `pml4_set_page()`가 PTE를 만들어 VA page와 frame을 연결한다.
  - QEMU 관점
    - PintOS의 guest physical memory는 QEMU host process의 memory region으로 backing된다.
  - 숫자로 확인
    - `VA = 0x8048123`
    - `page size = 0x1000`
    - `offset = 0x123`
    - `page base = 0x8048000`
- Q5. 같은 메모리 바이트가 왜 여러 의미를 갖는가?
  - 먼저 볼 문서
    - [[바이트-버퍼와-캐스팅-실험|바이트 버퍼와 캐스팅 실험]]
  - 핵심 키워드
    - raw byte
    - endian
    - pointer
    - integer
    - instruction
    - struct field
    - PTE bit
  - 실제 OS 관점
    - 커널 디버깅, crash dump, page table 분석은 모두 바이트를 타입으로 해석하는 일이다.
  - PintOS 관점
    - `struct thread`, `struct intr_frame`, `struct page`, `struct frame`을 주소에서 캐스팅해 관찰한다.
  - QEMU 관점
    - guest memory도 결국 QEMU가 관리하는 바이트 영역으로 읽고 쓴다.

## 코드 증거 지도

- syscall
  - PintOS 코드
    - `pintos/userprog/syscall.c`
      - `syscall_handler`
      - `f->R.rax`
      - `f->R.rdi`
      - `f->R.rsi`
      - `f->R.rdx`
  - 증명하는 것
    - syscall 의미는 QEMU가 아니라 PintOS가 해석한다.
    - register snapshot은 `struct intr_frame`으로 커널에 전달된다.
  - 연결 문서
    - [[syscall-end-to-end]]
- interrupt / scheduling
  - PintOS 코드
    - `pintos/devices/timer.c`
    - `pintos/threads/thread.c`
      - `thread_tick`
      - `schedule`
  - QEMU 코드
    - `hw/intc/apic.c`
      - `cpu_interrupt(cpu, CPU_INTERRUPT_HARD)`
  - 증명하는 것
    - PintOS는 interrupt handler에서 scheduling 결정을 예약한다.
    - QEMU는 guest CPU에 interrupt pending 상태를 전달한다.
  - 연결 문서
    - [[interrupt-timer-qemu]]
- address translation / page fault
  - PintOS 코드
    - `pintos/userprog/exception.c`
      - `page_fault`
      - `rcr2()`
    - `pintos/vm/vm.c`
      - `vm_try_handle_fault`
    - `pintos/threads/mmu.c`
      - `pml4_set_page`
  - QEMU 코드
    - `system/physmem.c`
    - `system/memory.c`
    - `target/i386/emulate/x86_mmu.c`
  - 증명하는 것
    - PintOS page table은 guest OS 자료구조다.
    - QEMU는 guest physical memory를 host process 안의 address space로 backing한다.
  - 연결 문서
    - [[address-translation-memory]]

## 숫자 계산 지도

- 주소 분해

    - `VA = 0x8048123`
    - `page size = 4096 = 0x1000`
  - 계산
    - offset = `0x123`
    - page base = `0x8048000`
    - virtual page number = `0x8048`
  - 연결 문서

- syscall 인자
  - 입력
    - `write(1, 0x8048123, 5)`
  - register
    - `RAX = SYS_WRITE`
    - `RDI = 1`
    - `RSI = 0x8048123`
    - `RDX = 5`
  - 연결 문서
    - [[syscall-end-to-end]]
- timer tick
  - 입력
    - `TIME_SLICE = 4`
  - 흐름
    - 1 tick
    - 2 ticks
    - 3 ticks
    - 4 ticks
      - `intr_yield_on_return`
  - 연결 문서
    - [[interrupt-timer-qemu]]
- page table 크기 감각
  - 입력
    - x86-64 4KB page
    - offset 12비트
    - 각 page table index 9비트
  - 의미
    - 각 단계 table은 512 entries
    - PTE 8B이면 table 하나는 512 * 8B = 4096B
  - 연결 문서
    - [[address-translation-memory]]
